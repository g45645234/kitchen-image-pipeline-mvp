# Host Reviewer Bridge

## Current MVP Handoff Status (2026-06-07)

This section records the current audit outcome and what has already been done.

### Implemented

- Three reviewer jobs are supported for each candidate: `codex`, `antigravity`, and `claude_cli`.
- Real reviewer jobs now require a downloaded candidate original. `POST /api/candidates/{candidate_id}/reviews/run` rejects candidates without `storage_key_original`, and the reviewer payload includes `image_file_path`, `image_file_available=true`, and `review_image_source=local_file`.
- Host reviewer bridge rewrites Docker storage paths through `HOST_STORAGE_ROOT` and refuses to invoke a reviewer CLI when the local image file is missing. Reviewer adapters instruct CLIs to inspect the local image file instead of judging from URL/metadata alone.
- Candidate reviewer results now include nullable `response_time_ms`, measured around each CLI invocation by both the in-app runner and the host reviewer bridge, and accepted/returned by the review API for observability.
- Review aggregation uses median score plus a `2-of-3` consensus/disputed rule. Reviewers do not auto-select final assets.
- Manual final selection is implemented in API and admin UI. Final selection requires usable rights (`may_use_directly=true`).
- Rights confirmation is implemented with operator comment/audit fields.
- Final asset rights confirmation is implemented at `POST /api/final-assets/{asset_id}/confirm-rights` and in the admin final panel. It is intentionally rights-only: it sets `manual_licensed`/`may_use_directly=true` with audit evidence but does not fabricate missing storage keys or files, so export-readiness still blocks broken legacy assets until storage is fixed or replaced.
- Operator upload of own final assets is implemented for wrong/right sides, including original, thumbnail, processed 1920x1080 JPEG, and metadata sidecar. Uploads are read with a configured size cap before image processing, so oversized files are rejected without loading unbounded data.
- Final asset listing/deletion now includes spec-compatible `GET /api/videos/{video_id}/final-assets?limit=100&offset=0` and `DELETE /api/final-assets/{asset_id}`. Delete is soft-delete plus audit event plus cleanup job; repeated delete after cleanup is idempotent and returns the original cleanup job instead of creating reruns/audit noise.
- Video and mistake lifecycle now includes soft-delete endpoints: `DELETE /api/videos/{video_id}` and `DELETE /api/mistakes/{mistake_id}`. Deleted rows are hidden from normal list/get routes, can be inspected with `include_deleted=true`, and enqueue `cleanup_storage` jobs with captured storage targets. Final selection and own-asset upload now reject deleted mistakes or videos.
- Audit coverage now includes the spec-required lifecycle events for manual MVP flows: `video.created`, `mistake.created`, `mistake.updated`, `candidate.approved_final`, `candidate.approved_reference`, `candidate.rejected`, `candidate.rights_confirmed`, `candidate.domain_blocked`, `final_asset.uploaded`, `final_asset.deleted`, `storage.cleanup_started`, `storage.cleanup_finished`, `storage.missing_file_detected`, `video.exported`, and `job.failed`. Legacy event names such as `rights_confirmed`, `domain_blocked`, and `reference_marked` remain written for backward compatibility. Export marks successfully exported final assets as `exported` only after manifest/CSV creation succeeds.
- Selected candidate final assets now generate final-asset-scoped thumbnail and `processed_1920x1080.jpg` files whenever a valid original file is present or downloaded during selection. Export now requires a processed file; if only a valid original exists, export generates derivatives first and then copies the processed 1920x1080 asset.
- Storage cleanup is implemented as dry-run/delete through API/UI and CLI (`python -m app.commands.cleanup_storage --dry-run|--delete`). Spec-compatible API aliases now exist at `POST /api/storage/cleanup-dry-run`, `POST /api/storage/cleanup`, and `POST /api/storage/cleanup?dry_run=true`, while the existing `/api/jobs/cleanup?dry_run=...` endpoint remains supported. Manual cleanup jobs remain global orphan cleanup; delete/replacement cleanup jobs run in targeted mode and delete only captured orphan keys.
- Idempotent active-job creation is implemented for search/export/cleanup jobs; completed/failed jobs can create explicit rerun jobs.
- Replacing a final asset through own upload or select-final now enqueues a targeted `cleanup_storage` job for old storage keys that are no longer referenced by the final asset. Select-final replacement also clears stale metadata sidecar references and EXIF flags before applying the new candidate.
- Candidate API listing now supports stable pagination through `skip` and `limit`.
- Admin candidate review page now supports spec-oriented filtering and pagination: `view`, `side`, `status`, `rights_status`, `source_provider`, stable sort values (`id`, `review_score`, `resolution`, `created_at`, with descending variants), `limit`, and `offset`. Candidate cards now expose operational metadata needed for review: status, storage status, rights status, direct-use flag, domain, quality/reference/review scores, and resolution. The `view=selected|consensus|disputed` filters are applied before count/offset/limit, so pagination totals and first page reflect the selected view instead of filtering only the current page.
- Search Query API now matches the spec flow: `POST /api/mistakes/{mistake_id}/generate-search-queries` creates idempotent generated `SearchQuery` rows, `GET /api/mistakes/{mistake_id}/search-queries` lists them with `side`/`limit`/`offset`, `POST /api/mistakes/{mistake_id}/search-queries` adds manual queries, `PATCH`/`DELETE /api/mistakes/{mistake_id}/search-queries/{query_id}` edits/removes manual or generated queries, and `POST /api/mistakes/{mistake_id}/search` creates per-query `run_search` jobs from existing SearchQuery rows, generating defaults only when no matching rows exist. Provider alias `mock` normalizes to `mock_search`. Admin UI now uses this spec flow: candidate pages can generate, manually add/edit/delete, and display SearchQuery rows before launching search, and video mistake rows call `/search` with JSON payload. The legacy `/api/mistakes/{mistake_id}/candidates/search` endpoint remains supported for backward compatibility and still enqueues `search_all_queries`.
- Spec-compatible list filters are implemented for the core read APIs: `GET /api/videos` supports `offset`, `limit`, and `status`; `GET /api/videos/{video_id}/mistakes` supports `offset`/`skip` plus `limit`; `GET /api/mistakes/{mistake_id}/candidates` supports `offset`/`skip`, `side`, `status`, `rights_status`, `source_provider`, and stable sort values (`id`, `review_score`, `resolution`, `created_at`, with descending variants).
- Video slugs are now validated at API schema level with a conservative lowercase letter/digit/single-hyphen pattern. Export and latest manifest/CSV lookup also derive a safe export slug from DB rows at use time, so legacy/imported unsafe slugs such as `../escaped`, `/tmp/escaped`, or `*` cannot move export packages outside `export_root` or glob-match another video's package.
- Video and mistake edit APIs are implemented: `PATCH /api/videos/{video_id}` updates title/slug/transcript/status with `video.updated` audit and rejects `status=deleted` outside the delete flow; `PATCH /api/mistakes/{mistake_id}` updates order, titles, timecodes, prompts, explanation, and negative criteria with `mistake.updated` audit. Both reject deleted rows, and mistake updates reject rows under deleted videos.
- Mistake creation now supports the spec path-scoped endpoint `POST /api/videos/{video_id}/mistakes` while keeping legacy `POST /api/mistakes` for compatibility. The admin video mistakes page now creates mistakes through the path-scoped endpoint instead of sending `video_id` in the JSON body.
- Admin UI now exposes manual edit/delete controls for video rows and mistake rows. The forms call the PATCH APIs with `actor=admin-ui`, delete buttons call the soft-delete endpoints with confirmation, and delete responses surface the queued cleanup job before reloading the page.
- Admin final-asset panel now runs a health check for selected final assets and shows warnings for legacy or broken records: non-exportable rights, bad storage status, missing/invalid storage keys, missing physical files, empty files, and missing processed assets. Links to `thumb`/`processed` are shown only when the corresponding file exists and is non-empty.
- A host-side Node smoke script now executes admin inline JavaScript from live UI HTML with mocked DOM/fetch/FormData (`scripts/smoke_admin_inline_js.mjs`). It covers create/edit/delete video and mistake flows, search, export, cleanup, upload final, run reviewers, confirm rights, select final, reference, reject, and block-domain fetch contracts. It is documented in `docs/admin_js_smoke.md` and remains a browserless smoke test, not a Playwright/Selenium replacement.
- Image downloading now fail-closes outside `APP_ENV=local` unless `ALLOWED_IMAGE_DOMAINS` is configured. In local mode, an empty allowlist keeps dev behavior open to any public http/https host; when configured, every image URL and redirect target must match an allowed host or subdomain before DNS/public-IP/MIME/size/image validation continues.
- `/api/reviewers/status` and the admin reviewer panel now reflect the host-bridge architecture explicitly: web no longer reports Docker-process CLI discovery as real reviewer readiness, and the response includes `execution_environment=host_bridge`, `ready=false`, and web-process diagnostics.
- Host reviewer bridge now writes an atomic heartbeat JSON status file (`HOST_REVIEWER_STATUS_PATH`, default `./storage/host_reviewer_bridge_status.json`). `/api/reviewers/status` reads that file with `HOST_REVIEWER_STATUS_TTL_SECONDS`; a reviewer reports `ready=true` only when the heartbeat is fresh and that reviewer is configured on the host bridge. The heartbeat contains pid/state/locked_by/reviewer configured booleans but not CLI command strings or tokens.
- Systemd handoff templates are available at `deploy/systemd/kitchen-host-reviewer-bridge.service.example` and `deploy/systemd/env.host-reviewer.example` for daemonized host bridge operation without mounting host CLI auth into Docker.
- Export now requires both `may_use_directly=true` and an exportable `rights_status` (`manual_licensed`, `own`, or `free_to_use`) so inconsistent unknown-rights data cannot leak into export. Export fails instead of creating an empty package when no assets meet the policy, includes manifest warnings for missing sides/mistakes, rejects absolute, parent-directory, empty-segment, and double-slash storage keys before reading or writing files, builds packages in a temporary directory before atomic rename so late copy/manifest/CSV failures do not expose partial export packages, and confirms export paths stay contained under `export_root`.
- Video export readiness is now visible before running export through `GET /api/videos/{video_id}/export-readiness` and the admin video mistakes page. The readiness check reports `can_export`, `complete`, active/ready mistake counts, exportable/ready asset counts, and blocking/non-blocking warnings. It mirrors the real export policy and treats a valid original without processed derivatives as export-ready with a warning, because export can generate missing derivatives before copying the processed file. `POST /api/videos/{video_id}/export` now enforces the same readiness gate and returns HTTP 400 without creating a job when `can_export=false`.
- The admin video mistakes page disables the `Export video` button when export readiness returns `can_export=false`, while still linking to the readiness JSON and showing the blocking warnings.
- Host reviewer bridge is hardened: `complete_job` and `fail_job` check `status='processing'` and `locked_by`, use `RETURNING id`, preserve the original exception if failure-state update also fails, and parse string booleans such as `force="false"` safely.
- Host reviewer bridge now refuses to run as root unless `HOST_REVIEWER_ALLOW_ROOT=true` or `--allow-root` is explicitly set. The systemd env template defaults `HOST_REVIEWER_ALLOW_ROOT=false`, and the service template documents that a dedicated CLI-auth owner is preferred.
- Host reviewer bridge has a no-credit `--check` preflight mode for systemd `ExecStartPre`. It validates DB connectivity, API `/health`, heartbeat path writability, and reviewer command env/executable presence without claiming jobs, writing heartbeat state, or invoking real reviewer CLIs.
- Host reviewer bridge now normalizes CLI outputs that are wrapped by a service envelope, including Claude Code `--output-format json` responses where the reviewer JSON is nested in the `result` field.
- The Docker worker remains filtered to non-CLI jobs: `extract_mistakes,create_search_queries,search_all_queries,run_search,score_candidates,review_candidate,create_reference_brief,download_candidate,process_final_asset,export_video,export_final_assets,cleanup_storage`. Real reviewer jobs stay on the host bridge.
- `videos.html` title block was fixed so the create-video JavaScript is no longer duplicated inside `<title>`.
- Admin/UI HTML smoke tests now cover rendered titles/scripts/forms/buttons, duplicate ids, unresolved Jinja markers, dead `href="#"` links, `/ui` versus `/admin` navigation prefixes, and a static JavaScript workflow matrix for UI actions: create video/mistake, search, export, cleanup, upload final asset, run reviewers, confirm rights, use as reference, create reference brief, reject candidate, block domain, and select final. The base header preserves the current prefix and offers a cross-link to the other alias.
- Admin auth is centralized in `app/auth.py`. In `APP_ENV=local` with no token the dev workflow remains open; when `ADMIN_API_TOKEN` is configured, project API/UI/job/storage routes require `X-Admin-Token` or an `admin_api_token` cookie; outside local env, protected routes fail closed with 503 if no token is configured.
- README now documents the current MVP startup and operator workflow: `docker compose up -d`, `alembic upgrade head`, `/admin`, `/docs`, manual video/mistake creation, SearchQuery generation/manual CRUD, mock search, candidate review with pagination, rights confirmation, own final upload, export readiness/export, storage cleanup, auth note, tests, and host reviewer bridge boundary.
- Candidate review API now includes spec-compatible `GET /api/candidates/{candidate_id}` and `POST /api/candidates/{candidate_id}/review` actions `approve_final`, `approve_reference`, and `reject`, while keeping legacy `approve` as an alias for `approve_final`. `approve_final` fails fast with HTTP 403 when `may_use_directly=false`, `approve_reference` is handled by the existing reference-only flow through the `review_candidate` job, and active review jobs are idempotent by candidate/action/reason/comment.
- Candidate cards now render spec-required per-candidate source metadata: `source_type`, optional `license_label`, and the exact `SearchQuery.query_text` for candidates linked through `query_id`. The admin route passes a `search_query_by_id` map to avoid template-side lazy loads/N+1 queries.
- Candidate status vocabulary now aligns with the MVP spec for new writes: final selection sets `ImageCandidate.status=approved_final`, reference marking sets `status=approved_reference` and `usage_role=reference_only`. API/admin status filters preserve backward compatibility by treating legacy `approved` as `approved_final` and legacy `reference_only` as `approved_reference`. FinalAsset statuses remain unchanged (`approved`/`exported`) because they are a separate model in the spec.
- `POST /api/videos/{video_id}/extract-mistakes` now enqueues an idempotent `extract_mistakes` job and returns `202 Accepted` with `JobResponse` instead of calling LLM/extraction inline in the HTTP request. The worker creates editable draft `Mistake` rows. Mistake extraction is `mock` by default even when `ANTHROPIC_API_KEY` is present; `MISTAKE_EXTRACTION_PROVIDER=anthropic` is required for external Anthropic quota use.
- Jobs API listing now supports the spec query shape `GET /api/jobs?status=running&limit=50&offset=0` while keeping legacy `skip`. Status filters include compatibility aliases: `running` matches internal `processing`, `succeeded` matches internal `completed`, and direct internal values still work. Response bodies still expose stored internal status values for backward compatibility.
- Worker job claiming now respects `settings.max_running_jobs`: before claiming a pending job, it requeues stale `processing` jobs, serializes PostgreSQL claimers with `pg_advisory_xact_lock`, counts active `processing`/`running` jobs, and returns without claiming when the limit is reached. `max_running_jobs <= 0` disables the cap. Stale processing requeue is ordered by oldest `locked_at` first.
- Image processing concurrency now respects `settings.max_image_processing_jobs`: job claiming counts active image-heavy jobs (`export_final_assets`, `review_candidate`) and skips new image-heavy jobs when the image limit is reached while still allowing light jobs such as `cleanup_storage`. Actual Pillow/download/derivative paths for upload, select-final, and export are guarded by a PostgreSQL advisory semaphore, with a process-local semaphore fallback outside PostgreSQL. Sync Pillow processing is run through `asyncio.to_thread` so it does not block the async event loop.
- Search rate limiting and provider error handling now align with the MVP spec: search request limits are capped at `50`, active `run_search` jobs are scoped to one job per `mistake_id`/`side`/`provider`, active rerun jobs with `:rerun:` suffixes are reused instead of duplicated, provider credential/network/HTTP/response errors mark `SearchQuery.status=failed` with classified error messages, single-provider search failures trigger job retry/failure, and multi-provider search jobs can finish as `partially_failed` while preserving successful results.
- Candidate quality scoring now follows MVP spec sections 18-19 for new search candidates: `score_quality = 0.50*resolution + 0.35*aspect_ratio + 0.15*source_metadata`, `review_score` defaults to `score_quality`, `score_visual` stays null, `reference_priority_score` stays null until manual reference marking, and rights are not part of the score. Low-resolution, poor-aspect, and incomplete-metadata results are stored with `quality_flags`/`is_low_quality` instead of being auto-rejected; exact duplicates remain deduped by `image_url_hash` through `ON CONFLICT DO NOTHING`.
- Blocked-domain auto-reject now covers both future and existing candidates. New search candidates are checked against `BlockedDomain` during upsert and saved as `status=auto_rejected`, `reject_reason=blocked_domain` when matched. Manual domain blocking normalizes the domain, rejects the selected candidate, and retroactively marks other review-queue candidates from the same domain as `auto_rejected` without changing already approved final/reference/rejected records.
- ReferenceBrief MVP flow is now implemented without external LLM quota use: reference-only candidates can enqueue `create_reference_brief` through `POST /api/candidates/{candidate_id}/reference-brief`, read the draft through `GET`, and manually edit it through `PATCH`. The worker creates deterministic draft briefs from `Mistake` prompts, negative criteria, candidate dimensions/domain, and source metadata. `important_visual_signs` and `do_not_copy` are JSONB arrays, and `error_message` is persisted/returned when brief generation fails so operators can diagnose failed async jobs. The admin candidate card shows a create-brief button for `approved_reference`/`reference_only` candidates. Real image generation from the brief remains post-MVP.
- Spec job vocabulary is now supported while preserving legacy compatibility: `create_search_queries`, `run_search`, `score_candidates`, `create_reference_brief`, `download_candidate`, `process_final_asset`, `cleanup_storage`, and `export_video` are supported by the worker. Existing `search_all_queries`, `review_candidate`, and legacy `export_final_assets` remain for compatibility. Upload now stores only the immutable original during the request and enqueues `process_final_asset` for thumbnail/1920x1080/metadata generation; select-final/export still auto-heal missing derivatives through the shared processing service. Standalone jobs are also recovery/maintenance entrypoints.

- Node/Python CLI dependencies do not have to be replicated inside the Docker image.

## Safety Notes

- If `ADMIN_API_TOKEN` is set in the app, protected API endpoints require `X-Admin-Token`; browser UI can authenticate with an `admin_api_token` cookie. The host bridge sends `X-Admin-Token` automatically when its `ADMIN_API_TOKEN` or `--admin-token` is set.
- The admin UI remains convenient in local/no-token mode. On a reachable server, set an admin cookie/session through the deployment layer or put `/ui` and `/admin` behind reverse-proxy auth.
- Reviewer CLIs receive the review payload through stdin; prompts and URLs are not interpolated into shell commands.

## Adapter Contract

`*_CLI_COMMAND` should normally call `python -m app.commands.reviewer_cli_adapter`.
